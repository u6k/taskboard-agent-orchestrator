# taskboard-agent-orchestrator
秘書AI・タスクボード仲介型AIエージェント実行基盤

## Redmine Webページブリーフィング要約・ブックマーク登録 v1

最初に実装するシナリオは、RedmineでAIユーザーが担当している「https://xxx をブリーフィング要約してブックマークに登録して。」のようなチケットを1件取得し、Webページ本文をブリーフィング要約してブックマークに登録する one-shot CLI です。

AIエージェントは、チケットを読み取ったら作業目的、成果物、作業計画、完了条件を整理してチケット詳細に記録します。その後、作業計画を1ステップずつ実行し、各ステップの完了ごとに計画へチェックを付け、コメントと成果物をチケットに残します。全ステップが完了したら、ステータスをレビュー中に更新し、担当者をチケット起票者へ戻します。

Webページ取得、本文抽出、ブックマーク登録はツールとして用意し、ブリーフィング要約はスキル内に記載されたプロンプトを使います。

### セットアップ

```powershell
uv sync
Copy-Item .env.example .env
```

`.env` に以下を設定します。実環境変数が同名で定義されている場合は、実環境変数が優先されます。

```dotenv
REDMINE_URL=https://redmine.example.com
REDMINE_API_KEY=replace-with-redmine-api-key
REDMINE_AI_USER_ID=123
OPENAI_API_KEY=replace-with-openai-api-key
OPENAI_MODEL=replace-with-openai-model
```

### 実行

```powershell
uv run taskboard-agent run-once
```

Redmineを更新せずに取得、計画生成、コメント生成だけ確認する場合:

```powershell
uv run taskboard-agent run-once --dry-run
```

基本ワークフローから分解した実装用ユースケースは [docs/use-cases.md](docs/use-cases.md) に整理する。

## 秘書AI・タスクボード仲介型 AIエージェント実行基盤

### 1. 企画概要

本企画は、秘書AIを人間のタスク管理窓口とし、タスクボードをAIエージェント群の実行管理基盤として位置づける業務運用基盤を構築するものである。

人間は、実施したい作業を秘書AIに依頼する。秘書AIは依頼内容を整理し、目的、期待する成果物、優先順位、期限、関連情報、必要な権限、完了条件などをタスクとして構造化したうえで、タスクボードに登録する。

タスクボードに登録されたタスクは、複数のAIエージェントが内容や担当領域に応じて取得し、必要なスキルを選択したうえで、スキルに定義された手順に従って実行する。進捗、成果物、判断待ち事項、必要な権限、作業履歴はタスクボード上で管理される。

AIエージェントが作業を進める中で、人間の判断、承認、権限付与、例外対応が必要になった場合は、チケットをレビュー中に戻し、判断待ち事項として記録する。秘書AIはその判断待ち事項を人間に分かりやすく要約し、何を判断すべきか、背景、選択肢、推奨案を提示する。人間が秘書AIに判断を伝えると、秘書AIがタスクボードを更新し、AIエージェントが作業を再開する。

これにより、人間はタスクボードを直接操作し続ける必要がなくなり、秘書AIとの対話を通じてタスク管理、優先順位調整、判断、承認、完了判定を行える。AIエージェント群は、タスクボードを起点として継続的に業務を遂行する。

---

### 2. 背景と課題

生成AIやAIエージェントの活用により、調査、開発、文書作成、事務処理、定型作業などをAIに任せられる可能性が広がっている。一方で、実際の業務にAIを組み込むには、単にAIにチャットで依頼するだけでも、人間が直接タスクボードを細かく運用するだけでも不十分である。

現状の課題は主に以下である。

* AIに依頼した作業の進捗が見えにくい
* 複数ステップにわたる作業を継続的に管理しづらい
* 人間がタスク登録、粒度調整、優先順位整理を毎回行う負担が大きい
* 人間判断が必要な場面で、何をすべきか分かりにくい
* AIが使うべきツールや権限を安全に管理する必要がある
* 定型業務をAIに反復実行させる仕組みが不足している
* AIの作業履歴や成果物を業務管理の中に残しにくい

これらを解決するには、人間とAIエージェントが直接やりとりするだけでなく、人間の依頼を業務タスクへ翻訳する秘書AIと、作業単位、進捗、担当、履歴を管理するタスクボードが必要である。

その中核となる流れは、**人間が秘書AIに依頼し、秘書AIがタスクボードを操作し、AIエージェント群がタスクボードからタスクを取得して実行する**ことである。

---

### 3. 目指す仕組み

本企画で目指すのは、**秘書AIを介して、人間がAIエージェント群に作業を依頼できる仕組み**である。

人間は、やってほしい作業を秘書AIに依頼する。秘書AIは依頼内容を確認し、不足情報があれば人間に質問する。依頼内容が実行可能な粒度になったら、秘書AIはタスクボードにタスクを登録する。タスクには、目的、期待する成果物、優先順位、期限、関連情報、必要な権限、完了条件などを設定する。

AIエージェント群は、タスクボードを監視し、自分が処理可能なタスクを取得する。取得したタスクは、タスク内容に応じて必要なスキルを選択し、各スキルに記述された作業手順を参照しながら小さなステップに分解して実行する。開発作業であればリポジトリや仮想環境を用意し、事務作業であれば安全な作業コンテキスト上で処理を行う。

AIエージェントが作業中に判断に迷った場合、必要なツールや権限が不足した場合、または人間の承認が必要な場合は、チケットをレビュー中に戻し、判断待ち事項を作成する。秘書AIは判断待ち事項を検知し、人間が判断しやすい形に整理して提示する。人間は秘書AIに判断、承認、権限付与、差し戻しなどを伝え、秘書AIがタスクボードを更新する。

人間は秘書AIに聞けば、AIが現在何を進めているのか、どこで止まっているのか、何を判断すればよいのかを確認できる。タスクボードは、秘書AIとAIエージェント群が共有する作業台帳として機能する。

---

### 4. 全体構成

本基盤は、以下の5要素で構成する。

#### 4.1 人間

人間は依頼者、判断者、承認者として関与する。

主な役割は以下である。

* 秘書AIへの作業依頼
* 優先順位や期限の意向提示
* 必要な権限の付与
* 判断・承認
* 例外対応
* 最終的な完了判定

人間はタスクボードを直接細かく操作するのではなく、秘書AIとの対話を通じて業務依頼と判断を行う。

#### 4.2 秘書AI

秘書AIは、人間の依頼をタスクボード上の実行可能なタスクへ変換するタスク管理担当である。

主な役割は以下である。

* 人間からの依頼受付
* 依頼内容の整理
* 不足情報の確認
* タスクの粒度調整
* 優先順位、期限、完了条件の整理
* タスクボードへの登録・更新
* 判断待ち事項の要約
* 選択肢と推奨案の提示
* 人間の判断結果の反映
* 完了報告の整理

秘書AIは、単なるチャット相手ではなく、人間の代わりにタスク管理を行い、タスクボードを操作する業務インターフェースとして機能する。

#### 4.3 タスクボード

タスクボードは、秘書AIとAIエージェント群が共有する作業台帳である。

主な管理対象は以下である。

* タスク内容
* 優先順位
* 担当AI
* ステータス
* 期限
* 関連情報
* 必要なツール・権限
* 作業履歴
* 判断待ち事項
* 成果物
* 完了条件

タスクボードは、単なるToDoリストではなく、AIが業務を実行するための実行管理基盤として機能する。

#### 4.4 AIエージェント群

AIエージェント群は、タスクボードに登録されたタスクを実行する実務担当である。

想定されるAIエージェントは以下である。

* 調査エージェント
* 開発エージェント
* テストエージェント
* 文書作成エージェント
* レビューエージェント
* 事務処理エージェント
* 定型業務エージェント

各AIエージェントは、タスクの取得、分解、実行、進捗更新、成果物作成、不明点のエスカレーションを行う。

#### 4.5 実行環境・ツール群

AIエージェントが実務を行うために、必要なツールや安全な作業環境を提供する。

主な要素は以下である。

* 外部システム接続
* API連携
* ファイル操作
* リポジトリ操作
* 仮想環境
* 権限管理
* 操作ログ
* 承認フロー

開発作業では、リポジトリを作成し、仮想環境を構築したうえで作業する。事務作業でも同様に、安全に分離された作業コンテキスト内で処理する。

### 5. 基本ワークフロー

本基盤における基本的な流れは以下である。

1. 人間が秘書AIに作業を依頼する
2. 秘書AIが依頼内容を整理し、不足情報を人間に確認する
3. 秘書AIが目的、成果物、優先順位、期限、完了条件をタスク化する
4. 秘書AIがタスクボードにタスクを登録する
5. AIエージェント群がタスクボードからタスクを取得する
6. AIエージェントがタスク内容から必要なスキルを選択する
7. AIエージェントがスキルに記述された手順を参照し、タスクを分解して実行計画を作成する
8. 必要なツール、権限、作業環境を確認する
9. AIエージェントがステップ単位で作業を実行する
10. 進捗や成果物をタスクボードに更新する
11. 人間判断が必要な場合は、チケットをレビュー中に戻し、判断待ち事項を作成する
12. 秘書AIが判断待ち事項を人間向けに要約し、選択肢と推奨案を提示する
13. 人間が秘書AIに判断・承認・権限付与を伝える
14. 秘書AIが判断結果をタスクボードに反映する
15. AIエージェントが作業を再開する
16. 作業完了時はチケットをレビュー中に戻し、成果物と作業履歴をタスクボードに提示する
17. 秘書AIが完了内容を人間に報告する
18. 人間が完了を確認し、秘書AIがタスクをクローズする

この流れにより、人間は秘書AIとの対話を中心に業務を依頼し、長い作業や複数ステップの作業はタスクボードを中心に継続的に管理できる。

#### 5.1 チケット状態遷移

チケット状態は、AIエージェントと人間のどちらが次に対応すべきかを明確にするために、以下の4状態で管理する。

* 未着手: 人間または秘書AIがタスクを登録し、AIエージェントがまだ取得していない状態。
* 進行中: AIエージェントがタスクを取得し、作業を進めている状態。人間が質問や判断に回答した後も、この状態に戻してAIエージェントが作業を再開する。
* レビュー中: AIエージェントから人間へ確認が戻っている状態。作業完了後の確認待ち、不明点の質問待ち、人間判断、承認、権限付与待ちを含む。
* 終了: 人間が成果物や作業内容を確認し、完了と判断した状態。

レビュー中は、完了確認待ちと質問・判断待ちの両方を表す。どちらであるかは、レビュー理由、確認事項、コメント、作業履歴で区別する。

基本的な状態遷移は以下である。

1. 人間または秘書AIがタスクを登録する: 未着手
2. AIエージェントがタスクを取得する: 未着手 -> 進行中
3. AIエージェントが作業を完了して人間へ戻す: 進行中 -> レビュー中
4. AIエージェントが不明点や人間判断を必要とする: 進行中 -> レビュー中
5. 人間がAIエージェントからの質問や判断事項に回答する: レビュー中 -> 進行中
6. 人間が完了と判断する: レビュー中 -> 終了

#### 5.2 ワークフロー例

基本ワークフローを具体的な業務に当てはめると、以下のような定型パターンとして扱える。

##### Webページブリーフィング要約・ブックマーク登録

最初に実装するシナリオとして、指示されたURLのページ本文を取得し、スキルで指定されたプロンプトに従ってブリーフィング要約を作成し、ブックマークサービスに登録する。

主な流れは以下である。

1. ユーザーが「https://xxx をブリーフィング要約してブックマークに登録して。」のようなチケットを起票する
2. AIエージェントがチケットを読み取り、作業目的、成果物、作業計画、完了条件を整理してチケット詳細に記録する
3. AIエージェントが作業計画を1ステップずつ実行する
4. Webページ取得ツールで対象URLへアクセスし、タイトル、本文、公開日、著者、主要リンクなどを取得する
5. 本文抽出ツールで広告、ナビゲーション、重複文などを除外し、要約対象の本文コンテキストを整理する
6. AIエージェントがスキル内に記載されたブリーフィング要約プロンプトを使い、要点、背景、示唆、注意点、次アクションを含む要約を作成する
7. ブックマーク登録ツールでURL、タイトル、要約、タグ、ブリーフィング要約への参照をブックマークサービスへ登録する
8. AIエージェントが各ステップ完了ごとに作業計画へチェックを付け、コメントを追加し、取得本文、抽出本文、ブリーフィング要約、ブックマーク登録結果を成果物としてチケットに残す
9. 作業計画がすべて完了したら、ステータスをレビュー中に更新し、担当者をチケット起票者に戻す

成果物は、取得元URL、ページメタデータ、抽出本文、ブリーフィング要約、ブックマーク登録結果、作業ログである。ログインや有料ページなど権限が必要な場合は、人間判断待ちとして扱う。

##### 文書考案

指示された内容をもとに、目的に合った文書を作成する。

主な流れは以下である。

1. 人間が秘書AIに文書の目的、対象読者、形式、含めたい内容、トーン、期限を伝える
2. 秘書AIが不足情報を確認し、文書作成タスクとしてタスクボードに登録する
3. 文書作成エージェントが構成案を作成し、必要に応じて人間確認待ちにする
4. 文書作成エージェントが本文ドラフトを作成する
5. レビューエージェントが目的との整合性、論理の流れ、表現、抜け漏れを確認する
6. 文書作成エージェントがレビュー結果を反映して最終案を作成する
7. AIエージェントが成果物、前提、未確認事項、差し戻し可能な論点をタスクボードに記録する
8. 秘書AIが文書案と確認ポイントを人間に提示する

成果物は、文書ドラフト、構成案、レビュー結果、確認ポイントである。事実確認が必要な内容や社外公開文書は、人間承認を完了条件に含める。

##### Obsidianに登録

指示された文書をObsidianに登録する。既存ノートがあれば追記または統合する。

主な流れは以下である。

1. 人間が秘書AIに登録したい文書、登録先Vault、ノート名やタグの希望を伝える
2. 秘書AIがObsidian登録タスクをタスクボードに登録する
3. ナレッジ整理エージェントがVault内の既存ノートを検索し、同一または関連するノートの有無を確認する
4. 既存ノートがない場合は、新規ノートとしてタイトル、本文、タグ、関連リンクを整えて登録する
5. 既存ノートがある場合は、重複を避けて追記・統合案を作成する
6. 既存内容の大幅な書き換えや削除が必要な場合は、人間判断待ちを作成する
7. AIエージェントが更新したノート、変更概要、関連ノート、未解決の統合判断をタスクボードに記録する
8. 秘書AIが登録先ノートと変更内容を人間に報告する

成果物は、Obsidianノート、新規作成または更新の差分、タグ、関連リンクである。既存ノートを破壊的に変更する操作は承認対象とする。

##### 開発

指示されたプロジェクトに対して、指示された機能をCodex CLIを使って開発する。

主な流れは以下である。

1. 人間が秘書AIに対象プロジェクト、実装したい機能、制約、期待する動作を伝える
2. 秘書AIが開発タスクをタスクボードに登録し、対象リポジトリ、ブランチ方針、完了条件を整理する
3. 開発エージェントがCodex CLIで作業環境を開き、既存コード、テスト、README、設定を確認する
4. 開発エージェントが実装方針を立て、必要なファイルを変更する
5. テストエージェントまたは開発エージェントが既存テスト、新規テスト、静的チェックを実行する
6. 失敗した場合は原因を調査し、修正して再実行する
7. 仕様不明点、危険な変更、外部サービス権限が必要な操作は、人間判断待ちとしてタスクボードに記録する
8. 開発エージェントが変更概要、テスト結果、残課題、ブランチやPRの情報をタスクボードに記録する
9. 秘書AIが実装結果と確認してほしい点を人間に報告する

成果物は、コード変更、テスト、実行結果、変更概要、必要に応じたブランチまたはPRである。破壊的なリポジトリ操作や本番環境への反映は、人間承認を必須にする。

##### 作業手順作成

指示されたことを実施するための作業手順書を作成する。主な利用者は人間である。

主な流れは以下である。

1. 人間が秘書AIに対象作業、前提条件、利用者、作業環境、完了状態を伝える
2. 秘書AIが作業手順作成タスクをタスクボードに登録する
3. 文書作成エージェントが作業を開始条件、準備、実施手順、確認手順、失敗時対応、完了報告に分解する
4. 必要に応じて調査エージェントが既存マニュアル、設定値、関連チケット、過去の作業履歴を確認する
5. 文書作成エージェントが人間が実行できる粒度で手順書を作成する
6. レビューエージェントが手順の順序、抜け漏れ、危険操作、ロールバック方法、確認観点を確認する
7. 人間承認が必要な危険操作や環境依存の値は、明示的な注意事項または判断待ちとして記録する
8. 秘書AIが手順書、前提条件、注意点、未確認事項を人間に提示する

成果物は、人間向け作業手順書、前提条件、チェックリスト、失敗時対応、確認観点である。手順書は、実行者がタスクボードを見なくても作業できる粒度を目標にする。

### 6. 必要な機能

#### 6.1 秘書AI機能

秘書AIには、人間の依頼を実行可能なタスクへ変換し、タスクボードを操作する機能が必要である。

必要な機能は以下である。

* 自然文依頼の受付
* 依頼内容の要約
* 不足情報の質問
* タスク粒度の調整
* 優先順位、期限、完了条件の整理
* タスクボードへの登録・更新
* 進捗状況の説明
* 判断待ち事項の要約
* 選択肢と推奨案の提示
* 人間の判断結果の反映
* 完了報告の作成

特に重要なのは、人間がタスク管理ツールを直接操作しなくても、秘書AIに依頼すればタスクが適切に作成・更新されることである。

#### 6.2 タスクボード機能

タスクボードには、通常のタスク管理に加えて、AI実行を前提とした情報管理が必要である。

必要な機能は以下である。

* タスク登録
* 優先順位管理
* ステータス管理
* 担当AI管理
* 秘書AIによる操作履歴管理
* レビュー中タスク管理
* 作業履歴管理
* 成果物管理
* 完了条件管理
* コメント・確認事項管理

特に重要なのは、秘書AIとAIエージェント群が同じタスク状態を参照し、作業に必要なコンテキストをタスクに紐づけて保持することである。

#### 6.3 AIタスク実行エージェント

AIエージェントは、タスクボード上のタスクを自律的に処理する。

必要な機能は以下である。

* タスクの自動取得
* タスク内容の理解
* 必要なスキルの選択
* スキルに定義された作業手順の参照
* 複数スキルを組み合わせた作業手順の分解
* ステップ単位の実行
* 進捗更新
* 成果物作成
* エラー報告
* 人間判断が必要な箇所の抽出
* 判断待ち事項の作成
* 利用したスキルと実行履歴の記録

AIエージェントは、単に回答を返すだけではなく、実際に作業を進める主体として振る舞う。

#### 6.4 スキル定義・選択機能

AIエージェントが作業を安定して実行できるように、本システムには複数のスキルをセットアップできる。スキルは、特定の作業を行うための作業手順、入力、出力、注意点、必要ツールを記述した定義である。

AIエージェントは、タスク内容を理解したうえで必要なスキルを自律的に選択し、選択したスキルに記述された手順に従って作業を進める。1つのタスクに対して複数のスキルが必要な場合は、各スキルの入出力をつなぎながら実行計画を作成する。

例えば、「xxxページを要約してu6k.Bookmarkに登録する」というタスクでは、AIエージェントは以下のようなスキルを選択して実行する。

* Webページ本文取得スキル
* 文章ブリーフィング要約スキル
* Bookmarkサービス操作スキル

スキルは、AIエージェントが参照すべき作業手順の単位であり、タスクそのものや定型業務テンプレートとは分離して管理する。

#### 6.5 Human-in-the-Loop機能

AIにすべてを任せるのではなく、人間が必要な場面で判断できる仕組みを用意する。

必要な機能は以下である。

* レビュー中タスクの表示
* 秘書AIによる判断内容の要約
* 承認依頼
* 権限付与依頼
* 例外対応依頼
* 選択肢と推奨案の提示
* 判断後のタスクボード更新
* 判断後のAI再開

ここでは、「人間に聞く」だけでは不十分である。秘書AIが、人間が判断しやすい形で論点を整理し、判断結果をタスクボードへ正しく反映する必要がある。

#### 6.6 ツール・権限管理

AIが外部システムを操作するには、安全な権限管理が必要である。

必要な機能は以下である。

* 利用可能ツールの定義
* システムごとの権限管理
* 最小権限での実行
* 権限不足時の確認
* 危険操作の承認
* 操作ログの記録
* 秘書AIを介した権限付与依頼

これにより、AIに作業を任せながらも、安全性と監査性を確保する。

#### 6.7 仮想作業環境

AIが作業するための安全な実行環境を提供する。

必要な機能は以下である。

* リポジトリ作成
* 仮想環境構築
* コード実行
* テスト実行
* ファイル編集
* 作業環境の分離
* 作業ログの保存
* 成果物の出力

開発作業だけでなく、資料作成や事務処理においても、作業コンテキストを分離して管理する。

#### 6.8 定型業務テンプレート

繰り返し発生する作業は、テンプレート化して秘書AIとAIエージェント群に任せる。

必要な機能は以下である。

* 定型業務の登録
* 入力項目の定義
* 実行手順の定義
* 承認ポイントの定義
* 担当AIの指定
* 実行履歴の管理
* 秘書AIによるテンプレート起動

これにより、ワークフローシステムに近い形で、定型業務をAIに継続実行させることができる。

#### 6.9 可視化ダッシュボード

人間と秘書AIがAIの作業状況を把握できるようにする。

必要な機能は以下である。

* 進行中タスク一覧
* AIごとの稼働状況
* レビュー中タスク
* ボトルネック表示
* 完了済みタスク履歴
* 成果物一覧
* エラー・停止中タスクの表示
* 秘書AI向けの要約情報

可視化により、AIの作業をブラックボックス化させず、秘書AIが人間へ状況説明できる状態にする。

### 8. 期待効果

本基盤により、以下の効果が期待できる。

#### 8.1 AIに作業を任せやすくなる

人間は秘書AIに作業を依頼すればよく、秘書AIがタスクボードに登録し、AIエージェント群がそれを取得して実行する。AIへの依頼が属人的なチャットではなく、管理可能なタスクとして扱われる。

#### 8.2 タスク管理の負担が下がる

人間がタスクの粒度、期限、完了条件、優先順位を毎回細かく入力しなくても、秘書AIが依頼内容を整理してタスクボードを更新する。

#### 8.3 進捗が見える化される

AIが何をしているのか、どこまで進んでいるのか、どこで止まっているのかをタスクボード上で確認できる。人間は秘書AIに状況を聞くことで、必要な情報を要約された形で把握できる。

#### 8.4 人間は重要判断に集中できる

作業の実行はAIエージェントに任せ、タスク管理は秘書AIに任せ、人間は優先順位、権限、承認、例外対応、完了判定に集中できる。

#### 8.5 長い作業を継続管理できる

複数ステップにわたる作業でも、タスクボード上に状態と履歴を残しながら進められる。秘書AIが進捗を整理することで、人間は長い作業の途中経過を把握しやすくなる。

#### 8.6 定型業務を自動化できる

繰り返し発生する作業をテンプレート化し、秘書AIが起動し、AIエージェントに反復実行させられる。

### 10. 段階的な実装案

#### フェーズ1：秘書AIによるタスク受付

まず、人間が秘書AIに依頼し、秘書AIが依頼内容を整理できる状態を作る。

主な実装内容は以下である。

* 自然文依頼の受付
* 依頼内容の要約
* 不足情報の確認
* タスク候補の生成
* 優先順位、期限、完了条件の整理

#### フェーズ2：タスクボード連携

次に、秘書AIがタスクボードを操作し、AIエージェントがタスクボードから内容を取得できる状態を作る。

主な実装内容は以下である。

* タスク登録
* ステータス管理
* 優先順位管理
* 秘書AI向けタスク操作API
* AI向けタスク取得API
* 作業履歴の記録

#### フェーズ3：スキルセットアップ

次に、AIエージェントが参照する作業手順としてスキルをセットアップできるようにする。

初期実装では、Webページをブリーフィング要約してブックマーク登録する手順をスキルとして定義し、ブリーフィング要約に使うプロンプトをスキル内に持たせる。

主な実装内容は以下である。

* スキル登録
* スキル内の作業手順定義
* スキルの入力、出力、注意点、必要ツールの定義
* AIエージェント向けスキル参照API
* スキル利用履歴の記録

#### フェーズ4：AIエージェントによるタスク実行

次に、AIエージェントがタスク内容に応じてスキルを選択し、スキル手順に従って実行できるようにする。

初期実装では、AIエージェントがWebページブリーフィング要約・ブックマーク登録チケットを読み取り、作業目的、成果物、作業計画、完了条件をチケット詳細に記録したうえで、計画を1ステップずつ実行する。各ステップ完了時には、作業計画のチェック、コメント追加、成果物記録を行う。

主な実装内容は以下である。

* スキル選択
* タスク分解
* 実行計画作成
* ステップ実行
* 進捗更新
* 成果物出力
* エラー時の報告

#### フェーズ5：人間判断フローの整備

AIが判断に迷った場合、秘書AIを通じて人間に適切に戻せるようにする。

主な実装内容は以下である。

* レビュー中への遷移
* 確認事項管理
* 承認依頼
* 権限付与依頼
* 秘書AIによる判断内容の要約
* 選択肢と推奨案の提示
* 判断結果のタスクボード反映
* 判断後のAI再開

#### フェーズ6：ツール・仮想環境連携

AIが実務を進めるための環境を整える。

初期実装では、Webページ取得、本文抽出、ブックマーク登録を個別ツールとして用意し、AIエージェントはスキル手順から必要なツールを呼び出す。

主な実装内容は以下である。

* 外部システム連携
* 権限管理
* リポジトリ操作
* 仮想環境構築
* 操作ログ管理
* 危険操作の承認

#### フェーズ7：定型業務テンプレート化

効果が確認できた業務をテンプレート化し、継続的な自動化に広げる。

主な実装内容は以下である。

* 定型業務テンプレート
* 実行条件の定義
* 承認ポイントの定義
* 担当AIの指定
* 秘書AIによるテンプレート起動
* 実行履歴管理

### 12. リスクと対策

#### リスク1：秘書AIが依頼内容を誤ってタスク化する

対策として、タスク登録前に目的、成果物、期限、完了条件を人間に確認できる仕組みを用意する。重要タスクでは、秘書AIが作成したタスク案を人間が承認してから登録する。

#### リスク2：AIの作業内容が見えなくなる

対策として、すべての進捗、判断、成果物、エラーをタスクボード上に記録し、秘書AIが人間向けに要約できるようにする。

#### リスク3：AIが誤った操作を行う

対策として、最小権限の原則を適用し、危険操作には人間承認を必須にする。権限付与や危険操作の承認は、秘書AIが論点を整理して人間に提示する。

#### リスク4：レビュー中で作業が止まる

対策として、レビュー中タスクの理由を明確に表示し、秘書AIが判断内容と選択肢を整理して提示する。

#### リスク5：タスクの粒度が粗すぎる

対策として、秘書AIが依頼段階でタスクを実行可能な単位に整理し、AIエージェントが必要に応じてサブタスクを作成する。

#### リスク6：定型化できない業務までAIに任せすぎる

対策として、初期は低リスクで手順が明確な業務から開始し、段階的に対象範囲を広げる。

### 13. まとめ

本企画で実現するのは、秘書AIが人間の依頼を受け付け、タスクボードを操作し、AIエージェント群がタスクボード上のタスクを取得して自律的に実務を進める仕組みである。

人間は秘書AIに作業を依頼し、優先順位や完了条件を伝える。秘書AIは依頼内容を整理し、タスクボードに登録する。AIエージェント群はタスクを取得し、必要なスキルを選択して、スキルに記述された作業手順に従って実行する。必要な判断や承認が発生した場合は、チケットをレビュー中に戻し、確認事項として記録する。人間の判断後、秘書AIがタスクボードを更新し、AIエージェントが作業を再開する。

この仕組みにより、AIへの依頼、タスク化、作業実行、進捗確認、人間判断、完了判定を一つの流れとして管理できる。

つまり、本基盤は、**秘書AIを人間のタスク管理窓口に置き、タスクボードをAIエージェント群の実行管理基盤にする、自律実行型タスク運用基盤**である。
