# ユースケース分解

READMEの基本ワークフローは、細かい処理単位ではなく、まず大きめのユースケースとして扱う。各ユースケースの中で、後からサブユースケースに分解して個別実装していく。

## UC-001: 秘書AIが依頼を受け付け、タスク化する

対応する旧UC: UC-001, UC-002, UC-003, UC-004

対応READMEフロー: 1, 2, 3, 4

人間が秘書AIに自然文で作業を依頼し、秘書AIが内容を整理して、タスクボードに登録可能なタスクへ変換する。不足情報があれば人間に確認し、目的、成果物、優先順位、期限、完了条件、必要権限を整理したうえでタスクを登録または更新する。

主なサブユースケース候補:

- 自然文依頼を受け付ける
- 依頼内容を要約する
- 不足情報を質問する
- タスク案を作る
- タスクボードへ登録する
- 既存タスクへ追記・更新する

最小実装:

- Redmineチケット本文を依頼入力として扱う
- LLMで理解メモ、目的、実施内容、完了条件、不明点を生成する
- Redmineコメントとして記録する

## UC-002: AIエージェントがタスクを取得し、計画・実行・進捗更新する

対応する旧UC: UC-005, UC-006, UC-007, UC-008

対応READMEフロー: 5, 6, 7, 8, 9, 10, 11

AIエージェントがタスクボードから処理対象タスクを取得し、内容を理解して実行計画を作る。必要なツール、権限、作業環境を確認し、実行できる範囲はステップ単位で進める。進捗、成果物、エラー、判断待ち事項はタスクボードへ記録する。

主なサブユースケース候補:

- AI担当タスクを1件取得する
- タスク内容を理解する
- 実行計画を作る
- 必要ツール・権限を確認する
- ステップ単位で作業する
- 進捗を更新する
- 成果物を記録する
- 判断待ち事項を作る
- 判断待ちを人間向けに要約する

最小実装:

- RedmineでAIユーザー担当の未終了チケットを1件取得する
- 実作業は行わず、理解メモまたは実行計画をコメントする
- 判断が必要な場合は確認待ちコメントを残し、人間側へ戻す

## UC-003: 人間の判断を反映し、AIエージェントが再開する

対応する旧UC: UC-009

対応READMEフロー: 12, 13, 14

人間が秘書AIに判断、承認、権限付与、差し戻しを伝え、秘書AIがその内容をタスクボードへ反映する。AIエージェントは更新された判断内容を読み取り、停止していた作業を再開する。

主なサブユースケース候補:

- 人間の回答を受け付ける
- 判断内容をタスクに記録する
- 判断待ち状態を解除する
- AI担当へ戻す
- 判断済み情報を読んで再開する

最小実装:

- 人間がRedmineコメントで回答する
- AIエージェントが次回実行時にコメント履歴を読み、再開可否を判断する

## UC-004: 成果物を提示し、完了報告する

対応する旧UC: UC-010

対応READMEフロー: 15, 16

AIエージェントが作業結果、成果物、作業履歴、未対応事項をタスクボードに記録する。秘書AIはそれを人間向けに整理し、完了報告として提示する。

主なサブユースケース候補:

- 成果物をタスクに紐づける
- 作業履歴をまとめる
- 未対応事項や制約を整理する
- 人間向け完了報告を作る

最小実装:

- Redmineコメントに完了報告を投稿する
- 成果物URL、ファイルパス、実施内容、確認してほしい点を記載する

## UC-005: 人間が完了確認し、タスクをクローズする

対応する旧UC: UC-011

対応READMEフロー: 17

人間が完了報告と成果物を確認し、問題なければ秘書AIがタスクをクローズする。差し戻しがある場合は、追加対応内容を記録して再度AIエージェントへ戻す。

主なサブユースケース候補:

- 完了可否を受け付ける
- タスクをクローズする
- 差し戻し内容を記録する
- 追加対応として再開する

最小実装:

- 人間がRedmineコメントで完了可否を記録する
- 秘書AIまたはAIエージェントがステータスを完了または対応中に更新する

## UC-006: 秘書AIが既存タスクを検索して提示する

追加ユースケース。

人間が「xxxをやるタスクってなんだっけ？」のように質問したとき、秘書AIが既存タスクを検索し、該当しそうなタスク一覧を提示する。人間はタスクボードを直接探さず、秘書AIとの会話で目的のタスクを見つけられる。

主なサブユースケース候補:

- 自然文の検索意図を解釈する
- キーワード、担当者、状態、期限などに変換する
- タスクボードを検索する
- 候補タスクを一覧表示する
- 関連度や状態を添えて提示する

最小実装:

- Redmineの件名・説明・コメントをキーワード検索する
- 候補としてチケット番号、件名、ステータス、担当者、更新日を表示する

## UC-007: 秘書AIがタスク状況を説明する

追加ユースケース。

人間が特定タスクについて「今どうなってる？」「何で止まってる？」「次に何をすればいい？」と聞いたとき、秘書AIがタスク内容、進捗、判断待ち、成果物、次アクションを説明する。

主なサブユースケース候補:

- タスクIDまたは自然文から対象タスクを特定する
- タスク本文、コメント、ステータス、担当者を読む
- 現在状況を要約する
- 停止理由や判断待ちを説明する
- 次に必要なアクションを提示する

最小実装:

- Redmineチケット1件を取得する
- コメント履歴を含めてLLMで状況説明を生成する

## UC-008: 秘書AIが人間の要求に応じてタスクを更新する

追加ユースケース。

人間が「期限を来週にして」「優先度を上げて」「担当をAIに戻して」「この条件を追加して」のように依頼したとき、秘書AIが対象タスクと更新内容を解釈し、タスクボードを更新する。

主なサブユースケース候補:

- 更新対象タスクを特定する
- 更新内容を構造化する
- 曖昧な場合は確認する
- タスクボードへ反映する
- 更新結果を人間へ報告する
- 危険または大きな変更では承認を求める

最小実装:

- RedmineチケットID指定の更新から始める
- コメント追加、担当者変更、ステータス変更、優先度変更のような限定操作だけ対応する

## 推奨する成長順

1. UC-002: AI担当チケットを1件取得し、理解メモまたは実行計画を残す
2. UC-007: 人間がタスク状況を聞けるようにする
3. UC-006: 既存タスクを検索して一覧提示できるようにする
4. UC-008: 人間の指示で限定的にタスク更新できるようにする
5. UC-003: 判断待ちからの再開フローを作る
6. UC-004: 成果物と完了報告を整える
7. UC-005: 完了確認とクローズを扱う
8. UC-001: 新規依頼受付からタスク登録まで広げる

## 前提

- 初期のタスクボードはRedmineとして扱う
- 最初は1回のCLI実行で1チケットだけ処理する
- 大きなユースケース内の細分化は、実装時にサブユースケースとして切り出す
- 初期段階ではAIの実作業より、理解、検索、説明、更新、判断待ちの運用を優先する
- 危険操作や外部システム操作は、人間承認フローが整うまで対象外にする
